func _handle_acceleration(delta: float, throttle_input: float, brake_input: bool) -> void:
	"""
	Handle forward/reverse acceleration and braking.
	
	Args:
		throttle_input: -1 to 1 (negative = reverse, positive = forward, 0 = coast)
		brake_input: true if brake button held
	"""
	if brake_input and current_speed > 0:
		# Braking while moving forward
		current_speed = move_toward(current_speed, 0.0, BRAKE_FORCE * delta)
	elif brake_input and abs(current_speed) < 0.1:
		# Brake held while stopped -> engage reverse
		current_speed = move_toward(current_speed, -REVERSE_SPEED, ACCELERATION * delta)
	elif throttle_input > 0:
		# Forward acceleration
		current_speed = move_toward(current_speed, MAX_SPEED, ACCELERATION * delta)
	elif throttle_input < 0:
		# Reverse input (alternative to brake-to-reverse)
		current_speed = move_toward(current_speed, -REVERSE_SPEED, ACCELERATION * delta)
	else:
		# No input - coast to stop
		current_speed = move_toward(current_speed, 0.0, DECELERATION * delta)
```

**Why this approach:**
- Uses `move_toward()` (same pattern as car.gd line 47) for smooth interpolation
- Forward acceleration: Ramps from 0 to MAX_SPEED over ~1.75 seconds (meets PHYS-03)
- Coast behavior: Gradual slowdown (not instant stop) for momentum feel
- Brake + reverse: Hold brake when stopped to reverse (intuitive for racing games)
- No speed loss in turns: `current_speed` independent of steering (meets PHYS-05 via CONTEXT.md "no speed loss through turns")

**Implementation notes:**
- `throttle_input` will come from ui_up (+1) / ui_down (-1) in _physics_process
- `brake_input` can use ui_down or separate brake action (user's choice)
- For now, using ui_down as both reverse and brake (standard kart racing control)

func _physics_process(delta: float) -> void:
	# Read input (reuse action names from car.gd lines 42-53)
	var steer_input = 0.0
	if Input.is_action_pressed("ui_left"):
		steer_input += 1.0
	if Input.is_action_pressed("ui_right"):
		steer_input -= 1.0
	
	var throttle = 0.0
	if Input.is_action_pressed("ui_up"):
		throttle += 1.0
	if Input.is_action_pressed("ui_down"):
		throttle -= 1.0
	
	var brake = Input.is_action_pressed("ui_down")
	
	# Handle steering
	_handle_steering(delta, steer_input)
	
	# Handle acceleration
	_handle_acceleration(delta, throttle, brake)
	
	# Calculate velocity from current_speed and forward direction
	var forward = -global_transform.basis.z  # Godot 3D forward is -Z (per AGENTS.md)
	velocity = forward * current_speed
	
	# Apply gravity if not grounded (temporary - Plan 02 will add proper ground detection)
	if not is_on_floor():
		velocity.y -= GRAVITY * delta
		is_grounded = false
	else:
		velocity.y = 0.0  # Stick to ground when grounded
		is_grounded = true
	
	# Move the kart (CharacterBody3D built-in method)
	move_and_slide()
```

**Key implementation details:**
- Input actions match car.gd exactly (lines 42-53) - same action names for compatibility
- Forward direction: `-basis.z` (per AGENTS.md: "negative Z is forward in Godot 3D")
- `velocity` is CharacterBody3D property - set directly, no force calculations
- `move_and_slide()` handles collision response automatically
- Temporary ground detection via `is_on_floor()` - Plan 02 replaces with raycast system
- Gravity applied when airborne - simple jump physics until Plan 02 adds proper air control

**Why move_and_slide():**
- Automatic collision handling (slides along walls instead of stopping)
- No configuration needed (just set velocity and call)
- Consistent behavior across platforms
